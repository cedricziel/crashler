## ADDED Requirements

### Requirement: User entity persisted in the database

The system SHALL persist user accounts as `App\Entity\User` rows in Postgres. Each user SHALL have a unique email (case-insensitively unique), a hashed password, a list of roles (stored as a JSON column), and a creation timestamp. The User class SHALL implement `Symfony\Component\Security\Core\User\UserInterface` and `PasswordAuthenticatedUserInterface`. The user identifier returned by `getUserIdentifier()` SHALL be the email.

#### Scenario: User persisted with email identifier
- **WHEN** an administrator creates a user with email `alice@example.com`
- **THEN** a row exists in the `user` table with that email
- **AND** logging in with that email + the original plaintext password succeeds
- **AND** the password column does NOT contain the plaintext

#### Scenario: Email uniqueness is case-insensitive
- **WHEN** a user `alice@example.com` already exists
- **AND** an attempt is made to create `Alice@Example.com`
- **THEN** the creation fails with a unique-constraint or validation error

#### Scenario: User implements Symfony security interfaces
- **WHEN** a User instance is loaded
- **THEN** `$user->getUserIdentifier()` returns the email
- **AND** `$user->getRoles()` returns at least `['ROLE_USER']`
- **AND** the entity is accepted by `UserPasswordHasherInterface::hashPassword`

### Requirement: Password hashing uses Symfony auto hasher

The system SHALL hash all user passwords using the Symfony `auto` password hasher (bcrypt or argon2id depending on availability). Plaintext passwords SHALL never be stored. The test environment SHALL keep the existing reduced-cost overrides (cost 4 for bcrypt; minimal time/memory cost for argon).

#### Scenario: Password is never stored in plaintext
- **WHEN** a user is created with password `correct horse battery staple`
- **THEN** the value persisted in the `password` column begins with `$2y$` (bcrypt) or `$argon2id$` (argon2id)
- **AND** the plaintext does not appear anywhere in the row

#### Scenario: Password verification succeeds with correct plaintext
- **WHEN** a login form posts the user's email + correct plaintext password
- **THEN** authentication succeeds and a session is established

### Requirement: Form-login authentication on /login

The system SHALL expose `/login` (GET, POST) and `/logout` (GET) endpoints powered by Symfony's form-login authenticator (`LoginFormAuthenticator`). The login form SHALL post `email`, `password`, and a CSRF token. On successful authentication, the user SHALL be redirected to the originally requested URL or to a configured default target. On failure, the system SHALL re-render the login form with an authentication-failure message and HTTP status as generated by Symfony's default flow.

#### Scenario: Anonymous request to /admin redirects to /login
- **WHEN** an unauthenticated request hits any URL under `/admin/`
- **THEN** the response is a 302 to `/login`
- **AND** the originally requested URL is preserved (Symfony's `_target_path`) so post-login the user lands back on it

#### Scenario: Login with valid credentials establishes a session
- **WHEN** a request POSTs `/login` with a valid email + password + CSRF token
- **THEN** the response is a 302 to the post-login target
- **AND** subsequent requests carry a session cookie that authenticates the user

#### Scenario: Logout clears the session
- **WHEN** an authenticated user requests `/logout`
- **THEN** the session is invalidated
- **AND** the response is a 302 to the configured logout target

### Requirement: Bootstrap admin via console command

The system SHALL provide a console command `crashler:user:create` that creates a User row from the command line. The command SHALL require `--email`, accept `--admin` to grant `ROLE_ADMIN`, and accept `--password` (or prompt interactively when STDIN is a TTY). The command SHALL refuse to run when STDIN is not a TTY and `--password` is absent. The command SHALL fail with a clear error if the email already exists; it SHALL NOT upsert.

#### Scenario: Create the first admin
- **WHEN** an operator runs `bin/console crashler:user:create --email=admin@example.com --admin --password=secret`
- **THEN** a User row is persisted with email `admin@example.com` and roles including `ROLE_ADMIN`
- **AND** the command exits 0
- **AND** the password column is hashed, not plaintext

#### Scenario: Email collision is rejected
- **WHEN** an operator runs the command with an email that already exists
- **THEN** the command exits non-zero with a "user already exists" message
- **AND** no row is inserted or updated

#### Scenario: Missing password without TTY is rejected
- **WHEN** the command runs with `--email` but no `--password` and STDIN is piped (no TTY)
- **THEN** the command exits non-zero with a clear error pointing to `--password`

### Requirement: New `admin` and `main` firewalls share the entity user provider

The system SHALL declare two new firewalls in addition to the existing `dev` and `ingest` firewalls:

- `admin` matches `^/admin` and gates EasyAdmin behind `ROLE_ADMIN` via `access_control`. It uses form login and an entity-backed user provider keyed on `User.email`.
- `main` is lazy and matches everything not caught by an earlier firewall. It also uses form login and the same entity user provider. The `main` firewall has no global role requirement; routes added in subsequent changes can apply per-route gates.

The `ingest` firewall (pattern `^/(v1|compat)/`) SHALL remain stateless and unchanged. The legacy `users_in_memory` provider SHALL be removed. The `IngestUserProvider` SHALL remain wired only to the `ingest` firewall.

#### Scenario: Firewall declaration order
- **WHEN** the application boots
- **THEN** firewalls are declared in this order: `dev`, `ingest`, `admin`, `main`
- **AND** a request to `/v1/logs` matches the `ingest` firewall and uses `IngestTokenAuthenticator`
- **AND** a request to `/admin/` matches the `admin` firewall
- **AND** a request to `/login` matches the `main` firewall (or `admin` if its pattern includes `/login`; firewall config must place `/login` outside `^/admin`)

#### Scenario: Non-admin authenticated user blocked from /admin
- **WHEN** a user with `ROLE_USER` (no `ROLE_ADMIN`) requests `/admin/`
- **THEN** the response status is 403

#### Scenario: Ingest path unaffected
- **WHEN** a request to `/v1/logs` arrives with `Authorization: Bearer <plaintext>` matching a configured token (DB or YAML)
- **THEN** the request is authenticated by `IngestTokenAuthenticator` exactly as today
- **AND** no session cookie is required, set, or read
