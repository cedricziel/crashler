## Context

Crashler today has zero persisted domain entities. `src/Entity/` is empty; the only Doctrine wiring in production paths is the messenger transport. The auth model is intentional and minimal: the `IngestTokenAuthenticator` hashes the incoming bearer token with SHA-256, looks the hex up in an in-memory `TenantRegistry` built once at boot from `crashler.yaml`, and attaches an `IngestUser` shim wrapping the matched `Tenant` value object. There are no logins, no sessions, no UI behind auth. The `tenants` spec encodes this minimalism literally: "SHALL NOT persist tenants in a database in this change. … SHALL NOT expose any HTTP endpoint, console command, or other runtime interface for creating, listing, modifying, or deleting tenants or their token hashes."

That cut was right for v1. The constraints it dodged — DB schema, password hashing, session storage, CSRF, admin authorization, audit trail, token rotation UX — would have eaten weeks for a system whose immediate job was "receive OTLP and write Parquet." Now the trade has flipped. The next milestone (a self-service UI, an invitation flow, a user-visible token-issuance flow) cannot be built on top of `crashler.yaml`. This change establishes the identity foundation those features will sit on, while keeping the existing ingest path bit-for-bit identical in behaviour.

## Goals / Non-Goals

**Goals:**
- Introduce User entity with email + hashed password, owning logins via Symfony's stock form-login authenticator.
- Introduce Org as a grouping unit; OrgMembership carries the (user, org, role) triple.
- Lift Tenant from a YAML-only config-bag into a full Doctrine entity with FK to Org, plus a separate per-Tenant membership and a TenantToken entity that finally has metadata (name, who created it, when, last-used).
- Provide a hybrid `TenantRegistry` so existing YAML tenants keep authenticating during the transition release, with a clear precedence rule.
- Provide a superadmin EasyAdmin UI for installation operators (the people who run `bin/console`) — distinct from the user-facing UI that ships in Change 2.
- Keep the ingest hot path's behaviour identical: `IngestTokenAuthenticator`, the 401 envelope, the constant-time hash comparison, the slug-as-path-component invariant.

**Non-Goals:**
- No public signup. Self-signup ships in Change 2 behind a config flag.
- No user-facing dashboard. Tenant owners cannot manage their own tenants/tokens until Change 2.
- No invitation flow. Tenant owners cannot invite teammates until Change 2.
- No OAuth, no SSO, no MFA. Email + password is the only auth mode in this change.
- No tenant-rename UX. The slug stays immutable post-creation; renaming is a separate (non-trivial) change because data on disk lives at `var/share/<signal>/<slug>/`.
- No tenant deletion that would orphan Parquet files. We block deletion with a guard rather than implement cascade-delete-of-disk-data; that is its own dangerous feature.
- No "switch to DB-only" yet. YAML stays supported in this change; deprecation is a follow-up after one release of observed dual-source operation.

## Decisions

### Decision 1: Email is the sole user identifier

**Choice:** `User.email` is the unique identifier returned by `getUserIdentifier()`. Stored lowercased on persist via a Doctrine lifecycle callback (or a `#[PrePersist]`); compared lowercased on login. No separate `username` column.

**Alternative considered:** username + email. Rejected — the operational story "I forgot my username" never recovers gracefully on a self-hosted ops tool, and email is what every real workflow (invitation, password reset later) keys on.

**Why:** matches the modern Symfony default flow generated by `make:user` when answering "yes" to "use email as identifier".

### Decision 2: Password hashing uses `auto`, not a pinned algorithm

**Choice:** keep the existing `password_hashers` config (`auto` → bcrypt by default; argon2id when sodium is available). The existing test override (cost 4 / argon time-cost 3) is preserved.

**Why:** `auto` is the Symfony recommended default and lets the framework pick the best available primitive without code changes. The current security.yaml already declares it.

### Decision 3: Org and Tenant slugs share rules; Tenant slug stays globally unique

**Choice:** Org slug and Tenant slug both match `^[a-z][a-z0-9-]{2,31}$` (no trailing hyphen). Org slug is unique within the `org` table. **Tenant slug is unique globally**, not just within its parent Org. This preserves the invariant that `var/share/<signal>/<slug>/` paths are unambiguous filesystem-wide.

**Alternative considered:** unique-per-org Tenant slug, with the filesystem path becoming `<org-slug>/<tenant-slug>/`. Rejected — it would invalidate every existing on-disk path and require a data migration. We may revisit this in a future "tenant rename / org reparenting" change.

**Why:** filesystem layout is a public contract; changing it is a separate decision from "where do tenants live in the schema."

### Decision 4: Two membership tables, not one polymorphic table

**Choice:** `org_membership` and `tenant_membership` are distinct tables. Both carry `(user_id, target_id, role)` but the FK targets are different and so is the lifecycle (org membership doesn't auto-grant tenant membership, and vice versa).

**Alternative considered:** single `membership` table with a `target_type` + `target_id` polymorphic association. Rejected — Doctrine doesn't support polymorphic FKs cleanly, and the two membership flows have genuinely different semantics (org admin can create tenants; tenant admin cannot create orgs).

**Why:** clearer schema, simpler queries, no polymorphic-FK acrobatics.

### Decision 5: Effective-tenant-access is the union of org-membership and tenant-membership

**Choice:** a user has access to a tenant if **either** they have an `OrgMembership` to the tenant's parent org **or** a direct `TenantMembership`. Roles compose by max precedence (`owner > admin > member`).

**Why:** matches the user's mental model from the discovery conversation. Lets an org owner manage all their tenants implicitly while still allowing tenant-only invites for outside collaborators.

### Decision 6: TenantToken keeps SHA-256, not bcrypt

**Choice:** stored token hash is lowercase 64-hex SHA-256 of the plaintext, exactly as the YAML form. Plaintext is generated server-side as `cw_<32-hex>`, shown once, never persisted.

**Alternative considered:** bcrypt/argon2id, treating tokens as passwords. Rejected — tokens have a different threat model (long, random, machine-only) and the existing ingest hot path does map-keyed lookup by hash. Switching to a slow KDF would force a per-request hash-each-row scan, which is unworkable.

**Why:** preserves the constant-time-lookup property of the current registry while finally giving tokens an audit trail. SHA-256 of a 128-bit-entropy random string remains brute-force-infeasible.

### Decision 7: Hybrid TenantRegistry composes DB-first, YAML-fallback

**Choice:** `TenantRegistry` becomes a thin facade over an ordered list of `TenantSource` providers. The default order is `[DbTenantSource, ConfigTenantSource]`. On hash collision between sources, the earlier source wins (DB beats YAML); within a source, duplicate hashes still hard-fail at boot. A short per-request memo cache fronts the lookup.

**Alternative considered:** import-on-startup-then-only-DB. Rejected — would force operators to migrate before deploying this change, which is exactly the friction we're trying to avoid.

**Alternative considered:** YAML-only with EasyAdmin writing back to YAML. Rejected — runtime YAML mutation conflicts with config caching, infrastructure-as-code workflows, and concurrent edits.

**Why:** lets operators adopt incrementally. A future change can flip the default to `[DbTenantSource]` only and emit a deprecation warning when ConfigTenantSource is non-empty; another follow-up can remove it entirely.

### Decision 8: `last_used_at` is updated post-response, not in-line

**Choice:** the auth path stays read-only. After the response is sent, a `kernel.terminate` listener (or a Symfony Messenger async dispatch — TBD by the implementing engineer) writes the new `last_used_at` to the matched token. Failures are logged at WARNING and never bubble.

**Why:** ingest latency is sensitive; we don't want to add a synchronous `UPDATE` to every authenticated request. The data is operational telemetry, not authentication state.

### Decision 9: Three firewalls, not two

**Choice:**

```yaml
firewalls:
  dev:    ^/(_profiler|_wdt|assets|build)/   # unchanged
  ingest: ^/(v1|compat)/                     # unchanged behaviour
  admin:  ^/admin                            # NEW — form login, ROLE_ADMIN
  main:   lazy: true                         # NEW — form login, public homepage, no global role
```

The `admin` firewall pattern matches before `main` because Symfony evaluates firewalls in declaration order. The `users_in_memory` provider is deleted; both authenticated firewalls share an `entity` provider (`App\Entity\User`, property `email`).

**Why:** keeping `admin` separate lets us tighten its access (e.g. IP allowlist, MFA later) without affecting the user-facing flows that Change 2 builds on `main`.

### Decision 10: Tenant deletion is blocked while data exists

**Choice:** a Doctrine event listener (or an EasyAdmin pre-delete hook) checks whether `var/share/<signal>/<tenant-slug>/` exists for any signal and rejects the delete with a clear error if so. Force-delete is not provided in this change.

**Alternative considered:** soft-delete (`deleted_at`). Rejected — soft-delete on the `tenant` row doesn't help if the goal is to free the slug for reuse, and the slug is the user-visible identifier.

**Why:** making accidental data loss require a real `rm -rf` is the right default. A "delete tenant + its data" UX is its own feature with its own confirmations.

### Decision 11: Bootstrap admin via console, not first-run web flow

**Choice:** `bin/console crashler:user:create --email=… --admin [--password=…]` creates the first admin. If `--password` is omitted, the command prompts interactively (hidden input) and refuses to run when STDIN is not a TTY without `--password`.

**Alternative considered:** "first request to the admin URL with no users in the database lets you create one." Rejected — too easy to misuse on a public install where the admin URL is reachable before anyone configures it.

**Why:** SSH access is the existing operator capability; piggybacking on it is the safer default.

### Decision 12: EasyAdmin, not hand-rolled CRUD

**Choice:** add `easycorp/easyadmin-bundle` and use it for the superadmin dashboard. We accept its conventions (CRUD controllers per entity, dashboard controller) and its UI defaults.

**Alternative considered:** roll our own. Rejected — superadmin CRUD is not where we want to differentiate. The user-facing UI in Change 2 will be hand-built precisely *because* it needs differentiation; the operator-only superadmin doesn't.

**Why:** ship value fast where the design isn't load-bearing.

## Risks / Trade-offs

- **Risk: ingest path now depends on Postgres availability.** Even with the per-request cache, a cold worker hitting a dead Postgres fails to assemble its registry → 401s for everyone. → Mitigation: the YAML fallback continues to authenticate during a DB outage as long as YAML tokens still exist. We document a "keep one YAML token per tenant for emergency continuity" runbook line in the README.
- **Risk: registry rebuild cost per request.** Loading every Tenant + TenantToken on every request is fine when there are dozens, dangerous at thousands. → Mitigation: a single SELECT joining `tenant` and `tenant_token` runs once per request; an APCu-backed second-tier cache keyed on a `(max(updated_at), count(*))` cache-buster can be added if and when the workload demands it. Out of scope here.
- **Risk: token plaintext leak via browser history / server logs.** EasyAdmin's URL after token creation must not contain the plaintext. → Mitigation: the plaintext is rendered into the response body of the redirect target, never into a URL or a flash message stored in session. The session does not retain the plaintext after the page is rendered once. Server-side log filters strip `Authorization` headers as today.
- **Risk: a user belonging to multiple orgs can leak tenant lists between orgs.** EasyAdmin shows everything to ROLE_ADMIN. The user-facing UI (Change 2) is where we'll need careful per-org scoping with Voters. → Mitigation: this risk doesn't materialise in Change 1 because only ROLE_ADMIN sees anything. Voters are Change 2's problem.
- **Risk: hash uniqueness across DB + YAML can drift.** A YAML hash and a DB hash might collide post-migration. → Mitigation: at boot, the registry computes the union and logs WARNING on cross-source duplicates; the DB entry wins per Decision 7. We explicitly do *not* hard-fail because the operator may be intentionally rotating from YAML to DB.
- **Trade-off: EasyAdmin's UI conventions.** It looks generic, not Crashler-branded. → Acceptable. The user-facing UI is where brand matters.

## Migration Plan

- **Roll-forward:**
  - Run `composer require easycorp/easyadmin-bundle`. (Symfony Flex pulls a recipe that registers the bundle and creates a stub `DashboardController` we'll replace.)
  - Run `bin/console make:user` answering: `User`, yes-store-in-db, `email`, yes-hash-password.
  - Run `bin/console make:auth` answering: empty authenticator → no, login form authenticator → yes, controller `SecurityController`, generate logout → yes.
  - Add the remaining entities (Org, Tenant, OrgMembership, TenantMembership, TenantToken) by hand or via `make:entity`.
  - Generate one consolidated migration: `bin/console make:migration` then trim/review by hand.
  - Deploy the migration via the existing deploy hook (`deploy.php` already runs `doctrine:migrations:migrate` if wired — confirm and add if missing).
  - Run `bin/console crashler:user:create --email=… --admin` on the production host.
  - Existing YAML tokens keep working unchanged; the operator can move tenants to DB at their own pace.
- **Rollback:**
  - The migration is reversible (Doctrine generates `down()`). Reverting also requires removing the EasyAdmin bundle and the firewall changes.
  - Any tokens created via EasyAdmin between deploy and rollback are lost (their plaintext was only ever shown once); operators must reissue them via YAML.
- **Communication:**
  - README gains an "Admin UI" section describing `/admin`, the bootstrap command, and the token-issuance flow.
  - The "Tenants and ingest tokens" section is updated to explain the hybrid model and recommend the DB path for new installs.
  - CHANGELOG-style notes (or commit-trailer convention) call out the breaking change to the `tenants` spec.

## Open Questions

- **EasyAdmin's audit trail.** EasyAdmin doesn't ship audit logging out of the box. Should creating/deleting a token be logged to a domain-events table (or even just to the structured log) as part of this change? Leaning *yes, structured log only*; full audit is its own change.
- **Token rotation UX.** Should tokens have a "rotate" action that issues a new plaintext and marks the old hash for deletion-after-N-hours, so clients can switch over without downtime? Likely yes, but it's UI surface that arguably belongs in Change 2 (the user-facing one) rather than EasyAdmin.
- **Membership invitation primitive.** Change 2 builds the invitation flow (email link → claim). Should the *primitive* (an `Invitation` entity with a tokenised one-time URL) live here as a half-built foundation, or fully in Change 2? Leaning *fully in Change 2* — adding it half-here would invite "what about invitations?" reviewer churn.
- **`citext` vs lowercased-on-write for `User.email`.** Postgres has `citext` for case-insensitive uniqueness. Doctrine's support for it is OK but not stellar. Lowercased-on-write + a unique index on the lowercased value is more portable and works with Sqlite for tests. Leaning *lowercased-on-write*; revisit if the team standardises on Postgres-only.
- **What does "owner" actually grant beyond "admin"?** In this change, `owner` and `admin` are functionally identical at both the org and tenant level (both can do everything via EasyAdmin since EasyAdmin is ROLE_ADMIN-only). The role becomes meaningful in Change 2 when Voters consult it. We're encoding the role now to avoid a schema migration in Change 2.
